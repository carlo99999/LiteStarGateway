Seventh pass, run right after Round 6's remediation landed (all 18 actionable R6 findings
fixed; only docs/mkdocs commits since). Nothing new shipped since the last pass, so this
round is deliberately a **fresh-eyes sweep of the whole tree** — five reviewers with

ops/config/tests) hunting for what six rounds missed, each cross-checked against Rounds 1–6

see H24's note).

**Baseline is the healthiest it has been:** full suite green (**622 passed**), overall
coverage 95%, `ruff` and `pyrefly` clean, no secrets tracked in git, all Round 6 fixes
re-verified intact in current source (SSRF guard incl. DNS-rebind re-resolution, SCIM admin
identity guard, `DECISIONS_READ` split, SSO audit trail, webhook token encryption).

**What Round 7 surfaces is a functional-correctness theme, not a security one:** for the
first time, no CRITICAL and no new authz/tenancy hole. The three HIGHs are all "the gateway
quietly does the wrong thing on a legitimate request": smart routing's judge/embeddings
calls consume real provider tokens that are never billed or budget-gated, three providers
silently drop `tools` and vision content while returning a confident-looking answer, and
streaming error translation never actually reaches the client at stream start (an intent the
code comments claim but only half-implement). The MEDIUMs cluster around race/constraint
handling the codebase already does correctly elsewhere but didn't apply everywhere, plus a
handful of prod-vs-dev divergences (SQLite FK enforcement, CI never touching Postgres, a
justfile recipe that re-introduces a fixed vuln).

New counts: **0 CRITICAL · 3 HIGH · 10 MEDIUM · 6 LOW**.

## HIGH

### R7-H22 — Judge/embeddings routing strategies make real, billable provider calls that bypass `UsageMeter` entirely

- **Where:** `application/routing/service.py:259-298` (`_embed_texts` → `self._gateway.aembeddings(...)`, `_judge_complete` → `self._gateway.achat_completion(...)`), invoked from `application/routing/judge.py` and `embeddings.py` for both the active and shadow strategies.
- **Issue:** `RouterService.route()` runs before `CompletionService._prepare` calls `UsageMeter.admit()` for the user's actual model. When a router uses the `judge`, `hybrid` (judge escalation), or `embeddings` strategy, the strategy calls the provider **directly through `LLMGateway`** with the team's real credential — consuming real tokens and real provider spend — but that call never touches `admit`/`settle_ok`/`_record_usage`. The cost is never written to `usage_event`, never counted against the team budget window, never traced. Two concrete consequences: a reproducible revenue leak on every routed request (the judge model call is pure untracked cost), and — because `admit()` never sees it — the judge call **runs even when the team is already over budget**, so a hard budget cap does not actually stop provider spend for teams with judge/embeddings routing enabled. **Verified:** both helpers read directly; no meter reference anywhere in `application/routing/`.
- **Fix:** Route judge/embeddings strategy calls through `UsageMeter` with a lightweight `admit`+`settle_ok` pair under a dedicated operation label (e.g. `routing.judge` / `routing.embeddings`) so they are budget-gated and billed like any other call; assert in a test that a judge-routed request writes a usage event for the judge model and is rejected when the team is over budget.

### R7-H23 — `tools`/`tool_choice` and vision content are accepted by the request allowlist but silently dropped for Anthropic, Vertex and Bedrock

- **Where:** `domain/request_policy.py:20-60` (allowlist admits `tools`, `tool_choice`, `parallel_tool_calls` for every provider) vs `infrastructure/llm/anthropic_adapter.py` (`kwargs["tools"]` set **only** in the structured-output emulation path, `:90-97`), `bedrock_adapter.py` (`toolConfig` only for structured output, `:122-123`), `vertex_adapter.py` (no tools translation at all); the `_text()` helpers in all three extract only `{"type": "text"}` parts, dropping `image_url` blocks.
- **Issue:** A client sending real function-calling (`tools`/`tool_choice`) to a model backed by one of these three providers gets a plain-text completion with `finish_reason: "stop"` — never `"tool_calls"`, never an error — because the translators never read `request["tools"]`. A client sending an image in message content gets it silently stripped and the model answers from the text alone. The allowlist explicitly blesses these fields, so the client has every reason to believe they work; the failure is invisible (no 4xx/501, no signal in the response). This is the same "silent capability lie" class R6-H17 fixed for routing fallback, but with no capability check anywhere on the direct call path. OpenAI/Azure are unaffected (they splat the sanitized request into the SDK).
- **Fix:** Raise `UnsupportedOperation` (501, the codebase's existing pattern for the embeddings/images provider matrix) when `tools`/`tool_choice` or non-text content parts are present in a request bound for a provider whose translator does not support them — or implement the translation. Either way, add per-provider tests for a `tools` request and an image-content request.

### R7-H24 — Start-of-stream provider errors never reach the client as an HTTP status — for any provider — despite the code's stated intent; Vertex additionally emits a synthetic chunk before the provider is even called

- **Where:** `infrastructure/web/api_router/completions.py:59-61` (comment: *"Resolve eagerly so errors become a proper HTTP status before the SSE starts"*), `application/completion_service.py:173-190` (`open_chat_stream` awaits only model/credential resolution — the returned generator is lazy), `infrastructure/llm/gateway.py:107` (same: "eagerly" resolves only the adapter), `infrastructure/llm/errors.py:156-166` (`translate_stream` is a pure relay, no priming); `vertex_adapter.py:244-251` (synthetic role-delta chunk `yield`ed **before** `generate_content_stream` is awaited).
- **Issue:** No layer primes the first chunk, so the provider SDK call happens on the first iteration of the response generator — after Litestar has already committed `200 OK` + SSE headers. **Empirically verified with a probe test** (fake OpenAI and Vertex clients raising `httpx.ConnectError` at stream creation): both paths die with *"Exception caught after response started"* — the carefully-built `translate_upstream_error` mapping (401→502, 429, timeouts) can never reach a streaming client at stream start. A streaming client with a bad upstream credential or a rate-limited provider sees `200` + an aborted connection instead of a retryable status, breaking SDK retry/backoff — the exact failure class `errors.py` exists to prevent, and the one the R6-H21 fix addressed for *mid*-stream. Note: the original reviewer claim was that only Vertex was broken; the probe showed **all providers** are — filed accordingly. Vertex is still worse in one honest respect: it yields a fabricated `role: assistant` chunk before contacting the provider at all, so its clients receive a "stream started fine" data frame even when the provider call instantly fails, and any future first-chunk priming fix would be silently defeated for Vertex unless its ordering is also fixed.
- **Fix:** In `open_chat_stream`/`open_responses_stream`, prime the stream (`await anext(...)` on the metered iterator, then re-chain the first chunk) so pre-first-chunk provider errors raise before the controller returns the `ServerSentEvent` — making the existing comment true. Move Vertex's synthetic role chunk to after `generate_content_stream` succeeds, matching the other three adapters' ordering. Test: provider raising at stream creation must produce the mapped HTTP status, not a 200.

## MEDIUM

- **R7-M49 — SCIM is the only pre-authentication bearer-token surface with zero rate limiting** (`web/scim/router.py:31-49` — no `middleware=` on either router; grep-confirmed `build_auth_rate_limit`/`build_inference_rate_limit` are wired into `/login`, `/signup`, `/sso/*`, `/invites`, `/password-resets`, `/reset-password`, `/v1/*`, but never `/scim/v2/*`). Every `/scim/v2/*` request hashes the presented token and does a DB lookup (`provide_scim_actor` → `get_by_token_hash`) before anything else — an unauthenticated attacker can flood it with garbage tokens for a cheap DB-load DoS, the exact shape the M33 fix addressed for `/v1/*`. *Fix: add `build_auth_rate_limit().middleware` (or a dedicated SCIM limiter) to `create_scim_router()`, and for depth to `create_scim_tokens_router()`.*
- **R7-M50 — Budget reservation multiplies by client `n` for providers that never forward `n`** (`application/usage_meter.py:119-130` reserves `output_ceiling × n`, `n` client-controlled up to `MAX_N=8`; grep-verified `to_anthropic_request`/`to_gemini_request`/`to_converse_request` never read `n` — those providers always return exactly one completion). For Anthropic/Vertex/Bedrock the reservation overshoots real spend by up to 8×, spuriously tripping `BudgetExceeded` for teams nowhere near their cap, worst under concurrency. *Fix: clamp/reject `n>1` at `_prepare` time for providers whose translator ignores it, or stop multiplying for them — either way make reservation and actual behavior agree.*
- **R7-M51 — Fire-and-forget shadow-routing tasks are never drained on shutdown, racing engine disposal** (`application/routing/service.py:74,254-256` — `_SHADOW_TASKS` exists only to prevent GC; grep-confirmed no lifespan hook in `app.py` awaits or cancels them). On every deploy/restart under load, a shadow task still inside its own `session_maker()` unit of work races the SQLAlchemy plugin disposing the engine; the broad `except Exception` in `_run_shadow` swallows it, so shadow decisions/usage rows are silently lost with only a warning log. *Fix: add a lifespan participant that `asyncio.gather(*_SHADOW_TASKS, return_exceptions=True)` with a short timeout on shutdown, before the DB plugin tears down.*
- **R7-M52 — Check-then-insert unique-constraint races unhandled in three write paths the codebase already handles correctly elsewhere** (`application/model_service.py:61-64`+`persistence/model_repository.py` (unique `(team_id,name)`), `credential_service.py:42-45`+`credential_repository.py` (unique `name`), `team_service.py:212-231`+`membership_repository.py` (unique `(team_id,user_id)`) — grep-confirmed `IntegrityError` is caught only in `budget_repository.py`, `router_repository.py`, `user_repository.py`). Two concurrent creates of the same model/credential/membership (double-click, retried client) both pass the pre-check; the loser's `IntegrityError` propagates as an unhandled 500 instead of the domain 409. *Fix: wrap the three `add()` commits in `try/except IntegrityError` → rollback → domain error, mirroring `router_repository.add`.*
- **R7-M53 — Credential deletion has no in-use guard, and dev/prod diverge because SQLite FK enforcement is never enabled** (`application/credential_service.py:53-57` — delete after existence check only; `orm.py` `ModelRecord.credential_id` FK has no `ondelete`; grep-confirmed no `PRAGMA foreign_keys` anywhere in `src/`). On Postgres (prod), deleting an in-use credential raises an unhandled `IntegrityError` → 500. On SQLite (dev/test), aiosqlite doesn't enforce FKs without the pragma, so the same delete silently succeeds, leaving `model.credential_id` dangling — every request to those models then fails at call time with `CredentialNotFound`, days later, and the failure a developer sees locally is not the one ops sees in prod. *Fix: check for referencing models in `CredentialService.delete` and raise a domain `CredentialInUse` (409); separately enable `PRAGMA foreign_keys=ON` via a connect event listener so dev matches prod.*
- **R7-M54 — `api_key.created_by` FK is unindexed; key revocation on user deactivation full-scans the table** (`persistence/orm.py:475` — `team_id`/`prefix`/`key_hash` are all indexed, `created_by` is not; `repository.py:72-82` `revoke_personal_keys_for_user` filters on it, called from both admin-driven and SCIM-driven deprovisioning). Locking out a departing/compromised user's keys — the path that most needs to be fast during incident response — degrades linearly with total key count. *Fix: `index=True` on `created_by` (or a composite `(created_by, service_principal_id, revoked_at)`) + Alembic migration.*
- **R7-M55 — `routing_decision` has no composite index for its only real query shape** (`persistence/orm.py:208-229` — two independent single-column indexes on `team_id` and `router_name`; every read in `router_repository.py` (`list_decisions`, `distribution`, `savings`) filters `team_id AND router_name`, and `list_decisions` orders by `created_at DESC` with no index support). The table is written on **every routed request**, so the analytics/export path degrades steadily; Postgres must bitmap-AND two indexes then sort. *Fix: replace with one composite `(team_id, router_name, created_at)` index and drop the two singles, via migration.*
- **R7-M56 — `just serve` re-introduces the exact `--forwarded-allow-ips "*"` vulnerability H9 fixed in the Docker entrypoint** (`justfile:93-95`, commented "Serve like production" — hardcodes `"*"`, no env override; contrast `docker-entrypoint.sh:19` which defaults to `127.0.0.1` and reads `FORWARDED_ALLOW_IPS`). Anyone deploying per the justfile (bare-metal/systemd instead of Docker) lets any direct client forge `X-Forwarded-For`, bypassing per-IP rate limiting and spoofing audit-log IPs — the vuln Round 2 closed, resurrected in a sibling code path. *Fix: mirror the entrypoint (`${FORWARDED_ALLOW_IPS:-127.0.0.1}`), or relabel the recipe "dev only".*
- **R7-M57 — CI never runs against Postgres or executes Alembic migrations for real, though prod requires both** (`.github/workflows/ci.yml` — whole suite runs on in-process SQLite; every `postgresql://` in tests is a string assertion; `docs/ci.md` itself lists "Postgres in CI" as an open decision; `docker-entrypoint.sh` runs `alembic upgrade` on every deploy). A migration valid under SQLite's permissive typing but broken under Postgres (type/constraint/JSON semantics) is first discovered in production, during a deploy. *Fix: add a CI job with `services: postgres` that runs `alembic upgrade head` and at least a subset of the suite with `DATABASE_URL` pointed at it.*
- **R7-M58 — Titan embeddings fan-out and per-event stream pulls share the same 8-worker Bedrock executor** (`bedrock_adapter.py:38` `_TITAN_EMBED_FANOUT = 8` semaphore over `:58` `_BEDROCK_EXECUTOR` (`max_workers=8`) — one embeddings request with ≥8 texts can occupy **all** workers for the duration of its network round trips, during which every concurrent Bedrock chat stream's `_next_event` hop — other teams' requests — has no worker and stalls). This reproduces, one level down, the same cross-request contention R6-M45's dedicated pool was introduced to eliminate. *Fix: separate (or headroom-size) the pools — e.g. a small dedicated executor for embed fan-out, keeping the event-pull pool uncontended.*

## LOW

- **R7-L35 — Bedrock mid-stream error table covers 4 of the documented codes** (`infrastructure/llm/errors.py:57-62` — `_AWS_ERROR_CODE_STATUS` maps `InternalServerException`/`ModelStreamErrorException`/`ServiceUnavailableException`/`ValidationException`; `ModelTimeoutException`, `ModelNotReadyException`, `AccessDeniedException` still fall through to the opaque-500 path R6-H21 fixed for the first four). *Fix: extend the table with the remaining ConverseStream error codes.*
- **R7-L36 — OFFSET pagination orders by a non-unique column with no tiebreaker in six repos, though `UserRepository.list` does it right** (`membership_repository.py:41-51`, `model_repository.py:51-61`, `service_principal_repository.py:32-42`, `credential_repository.py:58-62`, `audit_repository.py:38-47`, `router_repository.py:143-147`; contrast `user_repository.py:154-161` `order_by(created_at, id)`). Ties (bulk inserts same millisecond) get DB-arbitrary order; paging across them can skip/duplicate rows — most relevant for the compliance-facing audit list. *Fix: append `id` to every pagination `order_by`.*
- **R7-L37 — The rotation and reconciler background *loops* have no lifecycle test** (`infrastructure/rotation.py:91-123`, `usage_reconciler.py:41-55` — `rotate_all`/`reconcile` logic is tested, but the `asyncio.create_task` startup, shutdown cancellation, and the "failure must not kill the loop" guard are not). A regression in the loop wiring of two money/security-critical jobs would pass CI. *Fix: one lifespan-level test per loop (short interval, assert it ticks and cancels cleanly; make one tick raise and assert the loop survives).*
- **R7-L38 — `MLflowTraceSink.write` (the real MLflow SDK integration) has no test at all** (`infrastructure/observability/mlflow_sink.py:31-60`, 0% of `write`/`_experiment` covered; `mlflow` is pinned `>=3.14` with no upper bound, so an SDK signature change breaks prod tracing with zero CI signal — `MlflowMetricsPublisher` already shows the fake-client test pattern to copy). *Fix: unit test with a fake `MlflowClient` asserting `start_trace`/`end_trace` calls and status mapping.*
- **R7-L39 — CI never builds the Docker image** (`.github/workflows/ci.yml` — `Dockerfile`/`docker-entrypoint.sh` are the documented prod path but no `docker build` + `/health` smoke test exists; a broken slim-stage or HEALTHCHECK regression first surfaces on deploy). *Fix: a `docker build` + `docker run` + `/health` curl job, optionally path-gated to Dockerfile/compose changes.*
- **R7-L40 — No automated dependency-update bot** (no `.github/dependabot.yml`/Renovate; `pip-audit` in CI only flags *known* CVEs against *current* pins, so a fixed CVE waits for a manual `uv lock --upgrade`). *Fix: weekly Dependabot config for the uv ecosystem.*
